🌳 Concept 5: The Heap Priority Queue Array (t_heap_node *
)What is it in simple words?The subject states that coders cannot simply scramble and grab a dongle based on random thread speed; they must obey an orderly queue. You must implement a Binary Min-Heap array.Think of a Min-Heap like a pyramid of numbers stored flat inside a standard C array (heap[0], heap[1], heap[2]). It follows a strict sorting rule: the parent node at index 0 must ALWAYS hold the highest priority element (the lowest priority number value).
For FIFO: A coder's priority value is a sequential ticket number (box->ticket_counter++) [source: 2, source: 4]. Arrived first = lower ticket value = higher priority.
For EDF: A coder's priority value is their absolute burnout deadline timestamp (last_compile_time + time_to_burnout). Closest to dying = lower timestamp value = higher priority.
How the Heap shifts items:
push_heap & heapify_up: When a coder gets blocked, it appends itself to the very bottom leaf of the array [source: 2, source: 6]. heapify_up compares this node to its parent using index math (i - 1) / 2 . If the new coder has a higher priority (lower value), they swap places, bubbling up the tree until the pyramid is balanced].
pop_heap & heapify_down: When a coder takes a dongle, they must be removed from the root position (index 0). To fill the void, pop_heap grabs the element from the very end of the array and moves it to index 0. Then, heapify_down compares it against its children using index math 2 * i + 1 and 2 * i + 2, sinking it down until the next highest-priority coder bubbles up to the root position at index 0
